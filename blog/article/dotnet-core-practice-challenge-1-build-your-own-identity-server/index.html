ASP.NET Core Practice Challenge 1 — Build Your Own Identity Server

ASP.NET Core Practice Challenge 1 — Build Your Own Identity Server

Goal

Design and implement a custom authentication and user management system using ASP.NET Core MVC or Razor Pages without relying on built-in identity frameworks (no ASP.NET Core Identity, OpenIddict, or external auth libraries).

1. Core Feature Requirements

1.1 User Registration System

You must implement:

  • User sign-up with:
    • Name
    • Email
    • Password
  • Validation rules:
    • Email must be unique
    • Password must be securely stored (hashed + salted)
  • After registration:
    • User account is created in a disabled/unverified state

1.2 Email Verification System

You must implement:

  • Email confirmation flow
  • Generate a secure verification token
  • Send confirmation email containing a verification link
  • User must confirm email before login is allowed
  • System must support:
    • Token validation
    • Token expiration handling (optional but recommended)

1.3 Welcome Email System

After successful email confirmation:

  • Send a welcome email to the user
  • Email should confirm account activation

1.4 Authentication (Login/Logout)

You must implement:

  • Login using:
    • Email + Password
  • Rules:
    • Only verified users can log in
    • Invalid credentials must be rejected
  • Session management:
    • Cookie-based authentication
  • Logout functionality:
    • Clear session/cookie

1.5 Password Reset System

You must implement:

  • “Forgot Password” flow:
    • User enters email
    • System generates reset token
    • Email is sent with reset link
  • Reset flow:
    • Validate token
    • Allow password update
    • Invalidate token after use
  • Security requirements:
    • Token must expire after a short time window

1.6 User Profile Management

Authenticated users must be able to:

  • View profile
  • Update:
    • Name
    • Job title
    • Avatar image

Constraints:

  • Only logged-in users can access profile endpoints
  • Users can only modify their own data

1.7 Avatar Upload System

You must implement:

  • File upload mechanism
  • Store uploaded images securely (file system or storage folder)
  • Associate uploaded avatar with user profile
  • Validate file type and size

2. Security Requirements (Critical)

You must implement security manually:

2.1 Password Security

  • Passwords must never be stored in plain text
  • Must use:
    • Salted hashing (e.g., PBKDF2, BCrypt, or equivalent)
  • Must support password verification

2.2 Authentication Security

  • Use cookie-based authentication
  • Cookies must be configured securely:
    • HttpOnly
    • Secure (if HTTPS)
    • SameSite policy

2.3 Token Security

For:

  • Email verification
  • Password reset

You must ensure:

  • Tokens are:
    • Random
    • Non-guessable
    • Time-limited (for reset flow at minimum)
  • Tokens are invalidated after use

2.4 Input Validation

  • Validate all incoming data:
    • Email format
    • Required fields
    • Password rules (basic strength recommended)

3. Data Structure Requirements

You must design your own database schema for:

  • Users
  • Email confirmation tokens
  • Password reset tokens (or embedded fields)
  • Optional audit fields:
    • CreatedAt
    • LastLogin
    • UpdatedAt

4. UI Requirements (MVC or Razor Pages)

You must implement pages for:

  • Registration page
  • Login page
  • Email confirmation page
  • Forgot password page
  • Reset password page
  • Profile page (authenticated only)

5. Backend API / Endpoint Requirements

You must expose endpoints for:

Authentication

  • Register
  • Login
  • Logout

Email flows

  • Confirm email
  • Resend confirmation (optional enhancement)

Password reset

  • Request reset
  • Reset password

Profile

  • Get profile
  • Update profile

6. Testing Requirements

6.1 Unit Tests

You must test:

  • Password hashing and verification logic
  • Token generation and validation logic
  • User creation logic (service layer)

6.2 Integration Tests

You must test:

  • Registration flow end-to-end
  • Login flow
  • Email confirmation flow
  • Password reset flow
  • Profile update flow

Use test server + in-memory database or test DB.

7. Architecture Requirements

You must demonstrate:

  • Separation of concerns:
    • Controllers / Pages
    • Services
    • Data access layer
  • No business logic inside UI layer
  • Reusable services for:
    • Email sending
    • Authentication
    • Token generation
    • Password hashing

8. Non-Functional Requirements

8.1 Reliability

  • Handle invalid tokens gracefully
  • Prevent duplicate registrations

8.2 Security hygiene

  • No sensitive data in logs
  • No plaintext passwords or tokens exposed

8.3 Maintainability

  • Code should be modular and testable

9. Optional (Bonus Challenges)

If you want to push further:

  • Account lockout after failed login attempts
  • Rate limiting login endpoint
  • Two-factor authentication (email-based OTP)
  • Refresh token system (session extension logic)
  • Audit logs (login history tracking)
  • Admin panel to manage users
